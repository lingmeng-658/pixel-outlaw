# 阶段 A：优化敌人争抢道具行为

## A1. 当前问题

第二关目前只指定一个主要抢夺者。枪手接近强化道具时，如果玩家也靠近道具，枪手会优先移动到玩家与道具之间并攻击；该拦截行为没有明确时间上限，因此可能长期停留在道具附近却不完成拾取。

当前最终拾取停留为 `enemyPickupHoldMs: 480`。这不是唯一根因，但在只有一个主要抢夺者时，最后的停留也略显拖沓。

这会导致：

- Ammo / Buckshot 强化枪手很少真正出现；
- 敌人升级树和后续强化射击逻辑难以被玩家体验；
- 道具争夺更像“枪手守住道具”，而不是“玩家阻止敌人完成升级”。

## A2. 本阶段目标

让单个主要抢夺者稳定完成以下闭环：

```txt
道具落地
→ 主要抢夺者获得标记
→ 枪手可短暂拦截玩家或发动一次攻击
→ 拦截窗口结束后必须直接争取拾取
→ 在道具旁短暂停留
→ 完成升级
```

玩家仍然可以通过抢先拾取或击杀抢夺者进行打断；系统不能保证敌人无条件成功，只需保证玩家不干预时敌人能够可靠升级。

## A3. 明确实现要求

### 1. 为枪手拦截增加有上限的窗口

在 `LEVEL_TWO_CONFIG.contestedPickup` 中增加具名配置：

```ts
gunslingerInterceptMs: 700
```

含义：枪手被指定为主要抢夺者后，最多在前 700ms 内执行“移动到玩家与道具之间并攻击”的拦截行为。

窗口到期后：

- 即使玩家仍在道具 150 像素内，枪手也不再反复进入拦截分支；
- 枪手完成当前已经开始的攻击后，应转为直接向道具移动；
- 不要强行打断已经开始的枪手攻击流程；
- 不要通过每帧随机决定是否继续拦截。

### 2. 缩短最终拾取停留

将：

```ts
enemyPickupHoldMs: 480
```

调整为：

```ts
enemyPickupHoldMs: 320
```

320ms 仍需保留清晰的“敌人正在拾取”时刻，并允许玩家在最后阶段击杀或抢先拾取，但不应形成长时间原地等待。

### 3. 使用显式、可清理的运行时状态

建议为当前主要抢夺者增加场景级明确状态，例如：

```ts
private contenderInterceptUntil = 0
```

具体命名可以按现有代码风格微调，但语义必须明确。

要求：

- 每次分配新的主要抢夺者时重新计算窗口；
- 原抢夺者死亡、玩家抢走道具、敌人完成拾取、道具清理、离开区域、关卡重开时清零；
- 新抢夺者接替时拥有自己的完整拦截窗口；
- 不能让上一个道具的状态泄漏到下一个道具。

### 4. 整理判断逻辑，但不大拆系统

当前 `updateContenderMovement()` 已经包含枪手拦截、靠近、拾取停留和移动等多层条件。本阶段允许提取一到两个小型 helper，例如：

```ts
shouldGunslingerInterceptPickup(...): boolean
```

或语义相近的函数。

要求：

- helper 只负责清晰的一项判断或动作；
- 不新建泛化过度的 AI 框架；
- 不把整个敌人或道具争夺系统移动到新模块；
- 不改变其他敌人的争夺规则。

## A4. 本阶段不做什么

严格不要处理：

- 不增加同时争抢道具的敌人数量，第二关仍只有一个主要抢夺者。
- 不实现完整敌人升级树数据结构。
- 不新增第三关或新的敌人形态。
- 不修改枪手普通走位、边界感知斜退或随机移动；该设计已记录，另行开发。
- 不修改枪手攻击冷却、预警、预测、Ammo / Buckshot 第二组弹道或全局错峰逻辑。
- 不修改冲锋者行为。
- 不修改道具出现进度、道具顺序、玩家拾取效果或敌人强化结果。
- 不修改第二关批次、敌人数、存活上限、生成间隔和完成奖励。
- 不修改 Town Road 入场流程；战斗触发线属于阶段 B。
- 不修改存档结构或检查点语义。
- 不大范围重构 `MainScene`。
- 不安装新依赖。

## A5. 验收标准

代码与逻辑：

1. 枪手主要抢夺者的拦截行为有明确 700ms 上限。
2. 拦截窗口结束后，玩家持续靠近道具也不能让枪手无限守在中间。
3. 枪手可以完成已经开始的攻击，随后直接前往道具。
4. 敌人进入拾取距离后，320ms 后完成拾取。
5. 玩家先拾取、敌人死亡、抢夺者替换、离区、重开和清场时，新增状态均正确清理。
6. Coffee / Dynamite 的近战抢夺行为保持不变。
7. Ammo / Buckshot 的强化结果和后续射击逻辑保持不变。
8. 第二关仍只有一个黄色 `!` 主要抢夺者。
9. 不出现悬空标记、失效引用、重复拾取或旧状态污染下一次争夺。

试玩重点：

1. 玩家不干预 Ammo / Buckshot 争夺时，枪手应在有限时间内完成强化。
2. 玩家站在道具附近时，枪手可以短暂封锁或攻击，但不能长期不拾取。
3. 玩家在枪手拾取前击杀它，新抢夺者能够接替并继续流程。
4. 玩家抢先拾取后，敌人不再获得强化，标记和状态正常清除。
5. 多次道具争夺之间互不污染。

# 阶段 B：添加第二关安全准备区与战斗触发线

## 执行权限

**当前禁止执行。**

只有阶段 A 已提交、用户试玩或 ChatGPT 代码审查完成，并收到新的明确指令后，才可开始本阶段。

## B1. 本阶段目标

进入 Town Road 后先进入不限时的安全准备阶段：

```txt
进入 Town Road
→ 显示区域和战斗入口提示
→ 玩家可停留、观察或返回第一关
→ 主动向北越过触发线
→ 第二关遭遇战只启动一次
```

该区域将来用于购买道具和关前强化，本阶段只建立触发结构，不实现商店。

## B2. 设计要求

1. 增加明确状态，例如：

```ts
private levelTwoCombatStarted = false
```

命名可微调，但必须与 `levelTwoCompleted`、`levelTwoEncounter` 的职责区分清楚。

2. `enterTownRoad()` 在第二关未完成时：

- 重置第二关运行时；
- 不立即调用 `levelTwoEncounter.start()`；
- 玩家出生在南侧安全区；
- 显示清晰的横向触发标记和 `MOVE NORTH TO ENGAGE` 或同等简洁提示。

3. 玩家向北越过具名阈值后：

- 清除触发标记；
- 将战斗状态设为已开始；
- 使用当前 Phaser 游戏时间启动 `levelTwoEncounter`；
- 给出一次简洁反馈，例如 `AMBUSH STARTED`；
- 同一次进入过程中不能重复启动。

4. 触发阈值放入 `LEVEL_TWO_CONFIG`，例如使用具名的 `combatTriggerY`，不要散落裸数字。

5. `updateLevelTwo()` 或其上层调用必须在战斗未开始时停止遭遇推进、道具计划和第二关完成判断。

6. 触发标记应有独立、可统一清理的对象管理；进入完成状态、离区、重开和场景清理时不能残留。

7. 暂停、Restart、Continue 到 Town Road 入口时均回到安全准备阶段。战斗一旦触发，玩家向南退回安全区不能停止或重新启动战斗。

8. 第二关已经完成时，不显示战斗触发线，也不重新启动遭遇。

## B3. 本阶段不做什么

- 不实现购买界面、商店 NPC、商品、价格或金币消费。
- 不加入固定倒计时或自动开战。
- 不修改敌人批次、数量、生成频率、战斗参数或争夺规则。
- 不修改存档版本和中途战场保存。
- 不实现第三关。
- 不顺手处理 New Game、重复奖励或其他独立状态 bug，除非当前改动直接暴露构建错误；发现时记录风险并停止扩大范围。
- 不进行大范围场景拆分。

## B4. 验收标准

1. 进入 Town Road 后可以无限停留，期间不生成敌人、不推进隐藏批次、不生成争夺道具。
2. 玩家主动向北越过触发线后才开始第一批敌人生成。
3. 战斗只启动一次；越线后返回南侧不会暂停或重启遭遇。
4. 未触发时可以正常返回 Dusty Outskirts。
5. Restart Level 和未完成第二关的 Continue 均回到 Town Road 安全准备阶段。
6. 第二关完成后重新进入不会显示触发线或重复战斗。
7. 区域标题、南侧返回出口、暂停、清场和现有第二关玩法保持正常。
8. `npm run build` 通过。

## B5. 完成流程

完成阶段 B 后运行：

```bash

git diff
git status
```

更新 `docs/AI_HANDOFF.md`，提交信息：

```txt

```

阶段 B 仍需单独提交和 push，不得与阶段 A squash 成一个功能提交。
